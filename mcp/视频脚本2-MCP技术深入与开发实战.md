## 第一部分：MCP核心原语深度解析 (0:45-5:30)

### "工具箱"思维理解MCP原语 (0:45-2:30)

**核心思路**：把MCP想象成一个智能工具箱，AI是使用者，MCP原语就是里面的各种工具。

**服务器原语：AI的"工具箱"**
- **Tools (干活的工具)**：螺丝刀、扳手、锤子 → 读文件、查数据库、调API
- **Resources (参考资料)**：说明书、参数表 → 配置文件、系统状态
- **Prompts (操作手册)**：标准作业指导书 → 代码审查模板、分析模板

**实际使用分布**：
- Tools (90%+)：日常主力，如天气查询、文件操作
- Resources (<20%)：背景信息，如系统配置
- Prompts (<5%)：特殊场景，如专业分析模板

**客户端原语：工具的"回调"能力**
如果说服务器原语是"AI能用的工具"，客户端原语就是"工具能反过来请求AI做的事"：
- **Sampling**：工具请求AI生成内容
- **Elicitation**：工具请求用户输入信息
- **Logging**：工具向系统发送日志

这样的双向设计，让MCP不只是AI调用工具，工具也能"回调"AI！

### AI工具选择的神奇机制 (2:30-5:30)

这里我要解密一个核心问题：**AI是如何从众多工具中精确选择的？**

**第一步：工具"自我介绍"**
MCP自动将每个工具转换为AI可理解的文本：
```
Tool: get_weather
Description: 获取指定位置的天气预报
Arguments:
- latitude: 纬度 (required)
- longitude: 经度 (required)
```

**第二步：AI分析决策**
系统将所有工具描述组合成System Prompt发给AI：
"你是助手，可使用这些工具...选择合适工具或直接回复..."

**第三步：智能执行**
- AI分析用户问题："萨克拉门托天气怎么样？"
- AI输出：`{"tool": "get_forecast", "arguments": {"latitude": 38.5816, "longitude": -121.4944}}`
- 系统执行工具，获取结果
- 结果+原问题重新发给AI生成自然回复

**关键洞察**：
- 基于Prompt Engineering，任何AI模型理论上都能用MCP
- Claude专门优化过，工具选择更准确
- 容错设计：无效调用自动跳过，不会中断对话

**第二步：HTTP请求封装**
```python
async def make_nws_request(url: str) -> dict[str, Any] | None:
    """异步HTTP请求，带完整错误处理"""
    headers = {
        "User-Agent": USER_AGENT,
        "Accept": "application/geo+json"
    }
    
    async with httpx.AsyncClient() as client:
        try:
            response = await client.get(url, headers=headers, timeout=30.0)
            response.raise_for_status()
            return response.json()
        except Exception:
            return None
```
