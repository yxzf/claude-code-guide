## 开场白 (0:00-0:30)
大家好！今天我们来聊聊一个改变AI工具开发格局的新协议 - **Model Context Protocol，简称MCP**。

你有没有遇到过这样的问题：
- 想让Claude读取你的本地文件，但它说"抱歉，我无法访问"
- 不同AI应用需要重复开发相同的工具
- 数据要上传到云端才能处理，担心隐私安全

如果这些痛点你都遇到过，那这个视频绝对值得你看完。我会用最通俗的方式，告诉你MCP是什么，它如何解决这些问题。

## 第一部分：什么是MCP (0:30-3:00)

### 引入问题场景
想象一下这个场景：你想让AI帮你整理桌面文件。

**传统方式的困扰**：
- Claude Desktop：需要手动上传文件
- ChatGPT：完全无法访问本地文件  
- 每个平台都要重新开发工具

这就像每个品牌的手机都要专门的充电器一样麻烦。

### MCP的核心理念
MCP就像是AI工具的"USB接口标准"。

**一个生动的比喻**：
还记得以前各种设备都有自己的接口吗？iPhone用Lightning，安卓用Type-C，老设备用Micro-USB...

现在想象一下，如果有一个统一标准，让所有AI应用都能使用相同的工具接口，会怎样？

这就是MCP要做的事情！

### MCP的四大价值
1. **生态统一**：一次开发，处处运行
2. **平台无关**：告别厂商锁定
3. **数据安全**：本地处理，隐私可控  
4. **标准化**：统一接口，降低复杂度

## 第二部分：MCP架构设计 (3:00-6:30)

### 参与者角色
让我用一个简单的图来解释：

```
你 → Claude Desktop → MCP协议 → 文件系统服务器
```

**三个角色**：
- **Host（宿主）**：Claude Desktop，协调者
- **Client（客户端）**：MCP客户端，翻译官
- **Server（服务器）**：文件服务器，执行者

### 双层架构
MCP采用了聪明的双层设计：

**传输层**：
- STDIO：本地进程通信
- HTTP：网络服务调用
- SSE：实时数据流

**数据层**：
- 统一的JSON-RPC消息格式
- 所有传输方式共用同一套协议

这就像邮政系统：不管你用快递、平邮还是EMS，信件格式都是一样的。

### 核心原语概述
MCP提供了6种核心能力，分为两大类：

**服务器原语**（服务器提供）：
- **Tools**：让AI执行操作（读文件、查数据库）
- **Resources**：提供上下文信息（配置、状态）
- **Prompts**：专业模板指导

**客户端原语**（客户端提供）：
- **Sampling**：AI模型生成内容
- **Elicitation**：用户交互
- **Logging**：日志记录

## 第四部分：MCP vs 传统方案 (10:00-12:00)

### 对比分析表格

| 特性 | 传统Function Call | MCP协议 |
|-----|------------------|---------|
| **开发复杂度** | 每个平台重复开发 | 一次开发，多平台复用 |

| **扩展性** | 平台绑定 | 标准化，易扩展 |
| **维护成本** | 高（多套代码） | 低（统一标准） |
